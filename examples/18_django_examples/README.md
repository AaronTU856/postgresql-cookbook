# Chapter 18: Django Integration

## Overview

Django developers benefit from understanding the SQL generated by ORM queries.
This chapter maps common Django patterns to PostgreSQL concepts using the
cookbook online-store schema.

## Key Concepts

- Django models map to PostgreSQL tables.
- QuerySets generate SQL.
- `select_related` maps to joins for foreign keys.
- `prefetch_related` avoids N+1 query patterns for related collections.
- `annotate` maps to aggregates.
- `transaction.atomic()` maps to database transactions.
- Django indexes should reflect real PostgreSQL query patterns.

## Learning Outcomes

- Read SQL behind common Django ORM patterns.
- Recognise N+1 query risks.
- Understand aggregation from a Django point of view.
- Connect `transaction.atomic()` to PostgreSQL transactions.
- Design indexes that support Django filters.
- Explain Django/PostgreSQL integration in interviews.

## When To Use The Feature

Use these patterns when building Django APIs, admin pages, reports, background
jobs, and data-heavy features backed by PostgreSQL.

## When NOT To Use The Feature

Do not bypass the ORM for every query. Use raw SQL when it is clearer, faster, or
needs PostgreSQL-specific features the ORM cannot express well.

## Syntax Overview

```python
Order.objects.select_related("user").filter(status="paid")
```

Equivalent SQL shape:

```sql
SELECT ...
FROM orders
INNER JOIN users ON users.id = orders.user_id
WHERE orders.status = 'paid';
```

## Best Practices

- Inspect generated SQL for slow QuerySets.
- Use `select_related` for single-valued relationships.
- Use `prefetch_related` for collections.
- Add indexes for common filters and ordering.
- Use `transaction.atomic()` for multi-step writes.
- Keep business-critical constraints in PostgreSQL too.

## Performance Considerations

The ORM is productive, but it can hide N+1 queries, unnecessary columns, and
missing indexes. Pair Django debugging tools with PostgreSQL `EXPLAIN`.

## Common Mistakes

- Trusting ORM code without inspecting generated SQL.
- Forgetting indexes on frequent filters.
- Using `select_related` and `prefetch_related` interchangeably.
- Doing multi-step writes without `transaction.atomic()`.
- Keeping all validation only in Django forms or serializers.

## Business Examples

- Paid order API endpoint.
- Customer order history.
- Product catalogue filtering.
- Customer value annotations.
- Checkout transaction patterns.

## Interview Tips

Strong Django candidates can explain both the ORM code and the SQL shape it
generates. They also know when to inspect, optimise, or drop to raw SQL.
